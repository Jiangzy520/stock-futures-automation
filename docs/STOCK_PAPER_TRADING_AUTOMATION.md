# 股票模拟盘自动化

这份说明只讲公开版里可以安全分享的工程思路，不展开真实策略、私有接口和生产环境参数。

## 目标

股票模拟盘自动化的核心不是“让信号一出现就直接下单”，而是把下面几层稳定串起来：

```text
统一信号输入 -> 信号过滤 -> 模拟账户状态 -> 下单决策 -> 订单回写 -> 页面监控
```

## 解决思路

### 1. 先把信号统一成标准结构

不管前面接的是哪一路输入，进入模拟盘层之前都先整理成统一字段，例如：

- `symbol`
- `trading_day`
- `signal_time`
- `signal_price`
- `name`
- `source`

这样执行层面对的就不是不同来源的原始格式，而是同一套稳定协议。

### 2. 给信号消费做去重和时效过滤

模拟盘自动化最容易出问题的地方，通常不是“不会下单”，而是“重复下单”。

所以进入执行桥之前，至少要做：

- 同一信号唯一键去重
- 超时信号过滤
- 交易日过滤
- 交易时段过滤

如果这些基础过滤不先做，后面账户状态再完整，也很容易乱。

### 3. 在执行前先读取模拟账户状态

下单前要先知道当前模拟账户处于什么状态，而不是只看最新一条信号。

执行层至少要读取：

- 当前持仓
- 可用资金
- 未完成委托
- 当日下单记录
- 每个标的的已处理状态

这样才能决定一条新信号到底该不该进入下单流程。

### 4. 把“是否下单”做成一组明确约束

公开版里不展开具体策略参数，但工程上一般会把是否下单拆成几个固定判断：

- 是否为新信号
- 是否还在允许时效内
- 是否已有持仓
- 是否已有未完成委托
- 是否超过当日次数限制
- 是否超过单标的或总资金限制

这些约束成立之后，再去算价格和数量。

### 5. 用模拟执行器代替真实成交环境

在模拟盘阶段，重点不是追求和真实柜台一模一样，而是先保证：

- 下单请求有记录
- 委托状态可追踪
- 成交结果可回写
- 持仓变化可复现

也就是说，模拟执行器更像一个“订单状态机”，把信号变成：

```text
待处理 -> 已提交 -> 已成交 / 已撤销 / 已过期
```

这一步做稳定后，后面接真实交易环境时，改动会小很多。

### 6. 把执行结果回写到页面和日志

如果模拟盘自动化只做下单，不做状态回写，后面排错会非常痛苦。

建议至少回写这些内容：

- 最新信号是否已消费
- 当前订单状态
- 当前持仓状态
- 最近错误信息
- 最近一次执行时间

这样网站就不只是展示层，而是模拟盘自动化的监控入口。

## 一个更实用的理解方式

股票模拟盘自动化真正解决的问题，不是“如何自动买一笔”，而是：

`如何把信号、账户、约束、订单和状态回写组成一条可重复运行的链路。`

只要这条链路稳定，后面无论你接入新的数据输入、替换页面，还是继续扩展到期货，工程成本都会低很多。
